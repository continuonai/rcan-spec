# rcan-spec v3.0 Design

**Date:** 2026-04-11
**Repo:** continuonai/rcan-spec
**Status:** Approved — pending implementation plan

---

## Goal

Advance the RCAN protocol from v2.3 to v3.0, canonicalizing EU AI Act compliance schemas as first-class protocol citizens. Add §23–§26 to the spec. Update §22 FRIA with benchmark integration and Annex III prerequisites. Bump npm package from v1.10.0 to v3.0.0. No backwards compatibility required — Firestore/Firebase data will be reset.

---

## Architecture

Option 1 (additive): keep §1–§22 structure intact, add §23–§26 as new pages. No restructuring of existing sections. Breaking changes are limited to three well-defined rules at the protocol layer.

## Section 1: Version Bump and Breaking Changes

Protocol advances from v2.3 → v3.0. Three breaking changes:

### Breaking Change 1: Mandatory `sig` field
All RCAN messages must carry a valid ML-DSA-65 or pqc-hybrid-v1 signature in the `sig` field. Messages without `sig` are rejected. Previously optional.

### Breaking Change 2: Ed25519-only profiles rejected
Single-algorithm Ed25519 profiles are no longer accepted. Only pqc-hybrid-v1 (Ed25519 + ML-DSA-65) or ML-DSA-65-only profiles are valid.

### Breaking Change 3: FRIA prerequisite for Annex III registration
`REGISTRY_REGISTER` messages for high-risk AI systems (any valid `annex_iii_basis`) must include `metadata.compliance.fria_ref` pointing to a signed `rcan-fria-v1` document. Registration is rejected without it.

Updated `REGISTRY_REGISTER` metadata block:
```json
{
  "metadata": {
    "rrn": "rrn://org/robot/model/id",
    "robot_name": "my-robot",
    "compliance": {
      "annex_iii_basis": "safety_component",
      "fria_ref": "fria-20260411-signed.json"
    }
  }
}
```

### Version strings
- `rcan_version: "3.0"` in all RCAN config YAML files (was `"2.3"`)
- npm package: `1.10.0` → `3.0.0`
- Site: update all hardcoded version references to `v3.0`

## Section 2: §23 Safety Benchmark Protocol

**File:** `src/pages/spec/section-23.astro`

Documents the `rcan-safety-benchmark-v1` schema — a machine-readable artifact proving safety-critical software paths meet their latency thresholds, required as evidence for EU AI Act notified body review.

### Four benchmarked paths

| Path | What is measured | Default P95 threshold |
|------|-----------------|----------------------|
| `estop` | `SafetyLayer.emergency_stop()` call → halt state confirmed | 100 ms |
| `bounds_check` | Motor command evaluated against all BoundsChecker limits | 5 ms |
| `confidence_gate` | Confidence value through `ConfidenceGateEnforcer.evaluate()` | 2 ms |
| `full_pipeline` | Command received → safety-cleared or blocked | 50 ms |

### JSON schema

```json
{
  "schema": "rcan-safety-benchmark-v1",
  "generated_at": "2026-04-11T09:00:00.000Z",
  "mode": "synthetic",
  "iterations": 20,
  "thresholds": {
    "estop_p95_ms": 100.0,
    "bounds_check_p95_ms": 5.0,
    "confidence_gate_p95_ms": 2.0,
    "full_pipeline_p95_ms": 50.0
  },
  "results": {
    "estop":          { "min_ms": 0.3, "mean_ms": 1.2, "p95_ms": 4.1, "p99_ms": 7.2, "max_ms": 9.8, "pass": true },
    "bounds_check":   { "min_ms": 0.1, "mean_ms": 0.4, "p95_ms": 0.9, "p99_ms": 1.1, "max_ms": 1.4, "pass": true },
    "confidence_gate":{ "min_ms": 0.05,"mean_ms": 0.1, "p95_ms": 0.3, "p99_ms": 0.4, "max_ms": 0.5, "pass": true },
    "full_pipeline":  { "min_ms": 0.4, "mean_ms": 1.8, "p95_ms": 5.2, "p99_ms": 8.1, "max_ms": 11.0,"pass": true }
  },
  "overall_pass": true
}
```

### Integration with FRIA
When `overall_pass: true`, the file is referenced in a FRIA document under `safety_benchmarks.ref`. Links to §22. Generated by `castor safety benchmark`.

## Section 3: §24–§26 New Compliance Pages

### §24 Instructions for Use (`rcan-ifu-v1`)

**File:** `src/pages/spec/section-24.astro`

Documents the EU AI Act Art. 13(3) IFU schema. Eight required content fields plus metadata:

- `schema`: `"rcan-ifu-v1"`
- `generated_at`: ISO 8601 UTC
- `art13_coverage`: list of all 8 field names
- `provider_identity`: rrn, rrn_uri, robot_name, provider_name, provider_contact, rcan_version, agent_provider, agent_model
- `intended_purpose`: description, annex_iii_basis, deployment_context
- `capabilities_and_limitations`: summary, known_limitations[]
- `accuracy_and_performance`: note (links to benchmark)
- `human_oversight_measures`: hitl_gates, estop, confidence_gates, override
- `known_risks_and_misuse`: foreseeable_misuse[], mitigations[]
- `expected_lifetime`: software_support, hardware_dependent, note
- `maintenance_requirements`: software_updates, conformance_checks, incident_logging

Generated by `castor ifu generate`. `annex_iii_basis` must be one of the 10 valid Annex III categories.

### §25 Post-Market Monitoring / Incidents (`rcan-incidents-v1`)

**File:** `src/pages/spec/section-25.astro`

Documents the EU AI Act Art. 72 incident log schema. Persistent append-only JSONL format.

**Entry schema:**
```json
{
  "id": "uuid-v4",
  "timestamp": "2026-04-11T09:00:00.000Z",
  "severity": "life_health | other",
  "category": "string",
  "description": "string",
  "system_state": {}
}
```

**Report schema** (`rcan-incidents-v1`): `schema`, `generated_at`, `total_incidents`, `incidents_by_severity`, `reporting_deadlines` (15 days for `life_health`, 90 days for `other`), `art72_note`, `incidents[]`.

Managed by `castor incidents record | list | report`.

### §26 EU Register Submission (`rcan-eu-register-v1`)

**File:** `src/pages/spec/section-26.astro`

Documents the EU AI Act Art. 49 submission package schema. Wraps a validated signed `rcan-fria-v1` document.

**Schema:**
```json
{
  "schema": "rcan-eu-register-v1",
  "generated_at": "2026-04-11T09:00:00.000Z",
  "fria_ref": "fria-signed-20260411.json",
  "provider": { "name": "...", "contact": "..." },
  "system": { "rrn": "...", "robot_name": "...", "rcan_version": "3.0" },
  "annex_iii_basis": "safety_component",
  "conformity_status": "declared",
  "submission_instructions": "..."
}
```

Requires a valid signed FRIA (`rcan-fria-v1`) as input. Generated by `castor eu-register`.

## Section 4: §22 FRIA Updates (Additive Only)

Three additions to the existing `section-22.astro`:

### 4a. `safety_benchmarks` block
Add a subsection showing how benchmark results inline in a FRIA document:
```json
"safety_benchmarks": {
  "ref": "safety-benchmark-20260411.json",
  "generated_at": "2026-04-11T09:00:00.000Z",
  "mode": "synthetic",
  "overall_pass": true,
  "results": { ... }
}
```
Links to §23 for full schema.

### 4b. Related documents table
Add a table listing companion artifacts that together constitute a complete Art. 43 compliance package:

| Document | Schema | Article | Generated by |
|----------|--------|---------|--------------|
| FRIA | `rcan-fria-v1` | Art. 43 | `castor fria generate` |
| Safety Benchmark | `rcan-safety-benchmark-v1` | Art. 15 | `castor safety benchmark` |
| Instructions for Use | `rcan-ifu-v1` | Art. 13 | `castor ifu generate` |
| Incident Log | `rcan-incidents-v1` | Art. 72 | `castor incidents` |
| EU Register Submission | `rcan-eu-register-v1` | Art. 49 | `castor eu-register` |

### 4c. Annex III prerequisite note
Document the v3.0 breaking change: `REGISTRY_REGISTER` for Annex III systems requires `metadata.compliance.fria_ref`. Show the updated metadata block example (same as Section 1 above).
