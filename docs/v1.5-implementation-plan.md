## Executive Summary

RCAN v1.4 is a solid protocol with Ed25519 signing, Protocol 66 safety invariants, mDNS discovery, JWT role-based access, and HMAC-chained audit. However, the gap analysis identified **12 CRITICAL/HIGH gaps** that make v1.4 unsafe for production deployment and **10 MEDIUM/LOW gaps** that degrade fleet operations or compliance posture.

**v1.5 ships 12 MUST gaps + 6 SHOULD gaps = 18 of 22 gaps addressed.** Four large-complexity, non-blocking gaps are deferred to v1.6.

### Hard Deadlines
- **August 2, 2026** — EU AI Act compliance (GAP-10: training data consent, GAP-21: audit export). These are MUST regardless of priority ordering.
- **Fleet safety blocker** — GAP-03 (replay attacks) and GAP-11 (QoS/ESTOP reliability) must land before any production deployment of >5 robots.

### Top 5 Priorities
1. **GAP-03** — Replay attack prevention (CRITICAL, 1 page spec, ~16h) — highest ROI
2. **GAP-12** — Message format versioning (HIGH, 1 page spec, ~12h) — enables safe fleet upgrades
3. **GAP-04** — Time synchronization (CRITICAL, 1 page spec, ~13h) — required by GAP-03
4. **GAP-08** — Cloud relay identity (HIGH, 1 page spec, ~13h) — audit trail integrity
5. **GAP-05** — Consent wire protocol (CRITICAL, 3 pages, ~44h) — multi-runtime interop

### Additional Findings (Fix Before v1.5 Ships)
The gap analysis surfaced two spec/implementation bugs that must be fixed in v1.5 regardless of gap scope:
- **MessageType enum mismatch** — task context defines one numbering (`DISCOVER=1, STATUS=2…`) but rcan-py uses another (`COMMAND=1, RESPONSE=2…`). The spec must canonicalize integer values before new types are added.
- **SPEC_VERSION drift** — `rcan-py/rcan/registry.py` sends `rcan_version: "1.2"` while `rcan-py/rcan/message.py` uses `SPEC_VERSION = "1.4"`. Both must read from a single source of truth.

### GAP-12: Message Format Versioning *(do this first — unblocks all others)*

**Spec section to add:** §3.5 — Protocol Version Compatibility  
**Spec text summary:** MAJOR.MINOR.PATCH versioning. Receivers MUST accept messages from same MAJOR, lower-or-equal MINOR. Reject MAJOR mismatch with `VERSION_INCOMPATIBLE`. Unknown fields on lower-version receivers → ignore (forward compat). Missing required new fields on higher-version receivers → apply documented defaults.

**rcan-py changes:**  
- `rcan/message.py`: Add `rcan_version` validation in `RCANMessage.from_dict()` as first check before any other field processing. Add `VERSION_INCOMPATIBLE` to `exceptions.py`.  
- `rcan/validate.py`: Add `validate_version_compat(sender_version: str, receiver_version: str) -> bool`.

**rcan-ts changes:**  
- `src/validate.ts`: Add `validateVersionCompat(senderVersion: string, receiverVersion: string): boolean`.  
- `src/errors.ts`: Add `VERSION_INCOMPATIBLE` error type.

**OpenCastor changes:**  
- `castor/api.py`: Validate incoming `rcan_version` at message ingestion; return HTTP 400 with `VERSION_INCOMPATIBLE` if MAJOR mismatch.

**RRF changes:** None.  
**opencastor-client changes:** Display `rcan_version` in robot detail view; warn if robot version < app's expected minimum.

**Test requirements:**  
- v1.5 receiver accepts v1.4 message with unknown fields ignored  
- v1.5 receiver rejects v2.0 message  
- v1.5 receiver applies defaults for missing `delegation_chain`  
- Mixed-version fleet integration test

**Estimated hours:** Spec 2h + rcan-py 3h + rcan-ts 3h + OpenCastor 2h + client 1h + tests 2h = **13h**

### GAP-03: Replay Attack Prevention

**Spec section to add:** §8.3 — Replay Attack Prevention  
**Spec text summary:** Reject messages where `timestamp` is >30s old (configurable 5–300s). Maintain sliding-window `msg_id` seen-set covering replay window; reject duplicates. Safety messages (type SAFETY=6): window reduced to 10s. Apply replay check BEFORE signature verification to prevent replay-storm DoS.

**rcan-py changes:**  
- `rcan/validate.py`: Add `ReplayCache` class (LRU dict with TTL), `check_replay(msg_id, timestamp, window_s) -> None | raises ReplayError`.  
- `rcan/node.py`: Integrate `ReplayCache` into message receive path; initialize with `security.replay_window_s` from config.  
- `rcan/exceptions.py`: Add `ReplayAttackError`.  
- New config fields: `security.replay_window_s` (default 30), `security.msg_id_cache_size` (default 10000).

**rcan-ts changes:**  
- `src/validate.ts`: Add `ReplayCache` class with sliding-window eviction.  
- `src/schema.ts`: Add `replayWindowS` config type.

**OpenCastor changes:**  
- `castor/api.py`: Integrate replay check at HTTP endpoint entry. Return `{"error": "REPLAY_DETECTED", "msg_id": "..."}` on violation.  
- `castor/rate_limiting.py`: Ensure replay check is separate from (and preceding) rate limiting so ESTOP replay is caught without being mistaken for rate limit.

**RRF changes:** None.  
**opencastor-client changes:** None (client-side; replay is a receiver concern).

**Test requirements:**  
- Replayed command (same msg_id, same timestamp) → rejected  
- Command with stale timestamp (31s old) → rejected  
- Valid command accepted  
- ESTOP with 9s-old timestamp → accepted; 11s-old → rejected  
- Replay cache correctly evicts entries outside window  
- ESTOP is never blocked by replay check when timestamp is fresh

**Estimated hours:** Spec 2h + rcan-py 4h + rcan-ts 3h + OpenCastor 3h + tests 3h = **15h**

### GAP-01: Command Delegation Chain

**Spec section to add:** §12 — Command Delegation and Chain of Custody  
**Spec text summary:** Commands sent by a robot on behalf of a human MUST carry `delegation_chain: DelegationHop[]`. Each hop: `{issuer_ruri, human_subject, timestamp, scope, signature}`, signed by issuing principal's private key. Receivers MUST verify all hop signatures and reject if `human_subject` lacks requested scope on target. Max depth 4 hops; reject with `DELEGATION_CHAIN_EXCEEDED` if deeper. CommitmentRecord MUST serialize full chain.

**rcan-py changes:**  
- `rcan/types.py`: Add `DelegationHop` dataclass; add `delegation_chain: List[DelegationHop]` to `RCANMessage`.  
- `rcan/signing.py`: Add `sign_delegation_hop(issuer_key, human_subject, scope, target_ruri) -> DelegationHop`; add `verify_delegation_chain(chain, trusted_keys) -> bool`.  
- `rcan/validate.py`: Add `validate_delegation_chain()` enforcing max depth and signature validity.  
- `rcan/exceptions.py`: Add `DelegationChainExceededError`, `DelegationVerificationError`.  
- `rcan/audit.py`: Extend `CommitmentRecord` to serialize `delegation_chain`.

**rcan-ts changes:**  
- `src/types.ts`: Add `DelegationHop` interface.  
- `src/crypto.ts`: Add `signDelegationHop()`, `verifyDelegationChain()`.  
- `src/validate.ts`: Add `validateDelegationChain()`.
- `src/audit.ts`: Extend `CommitmentRecord`.

**OpenCastor changes:**  
- `castor/command_interpreter.py`: When Robot A issues command to Robot B, build and attach `delegation_chain` using issuing human's cached auth context.  
- `castor/auth.py`: Validate incoming delegation chain against R2RAM scope table; reject non-compliant commands.  
- `castor/audit.py`: Ensure chain is written to commitment record.

**RRF changes:** Registry must store robot public keys for delegation chain verification. Add `GET /api/v1/robots/{rrn}/public-key` endpoint.

**opencastor-client changes:** Audit log view: display delegation chain as visual breadcrumb trail (Human A → Robot A → Robot B). Show each hop's scope and timestamp.

**Test requirements:**  
- Valid 2-hop chain: Robot A → Robot B on behalf of Human A  
- Chain exceeding depth 4 → rejected  
- Tampered hop signature → rejected  
- Missing chain when sender is robot acting on behalf of human → rejected  
- Audit record contains full chain

**Estimated hours:** Spec 4h + rcan-py 8h + rcan-ts 7h + OpenCastor 6h + RRF 2h + client 4h + tests 5h = **36h**

### GAP-02: Robot Identity Revocation

**Spec section to add:** §13 — Robot Identity Revocation  
**Spec text summary:** Registry MUST expose `GET /api/v1/robots/{rrn}/revocation-status` → `{status: "active"|"revoked"|"suspended", revoked_at, reason, authority}`. Robots check on startup; cache with 1h TTL. New `ROBOT_REVOCATION (19)` message type broadcast by registry. Peers receiving MUST invalidate all cached consent and public key material for revoked RRN. ESTOP valid from any source; RESUME from revoked robot → rejected. Offline: continue ≤ `max_revocation_staleness_s` (default 3600s) then quarantine.

- `rcan/node.py`: Check revocation on startup; add handler for `ROBOT_REVOCATION` broadcast; reject `RESUME` from revoked robot.

**rcan-ts changes:**  
- `src/types.ts`: Add `RevocationStatus` interface.  
- New `src/revocation.ts`: `RevocationCache`, `checkRevocationStatus()`, `handleRevocationBroadcast()`.

**OpenCastor changes:**  
- `castor/registry.py`: Add revocation status check on robot registration.  
- New `castor/revocation.py`: Broadcast `ROBOT_REVOCATION` message to all peers when registry marks robot revoked.  
- `castor/auth.py`: Block all commands from/to revoked robots (except ESTOP received).  
- `castor/offline_fallback.py`: Quarantine mode after `max_revocation_staleness_s`.

**RRF changes:**  
- Add `GET /api/v1/robots/{rrn}/revocation-status` endpoint.  
- Add `POST /api/v1/robots/{rrn}/revoke` (creator role required).  
- Add `revocation_status`, `revoked_at`, `revocation_reason` fields to robot Firestore document.

**opencastor-client changes:**  
- Robot detail screen: show revocation status badge (🟢 Active / 🔴 Revoked / 🟡 Suspended).  
- Owner actions: "Suspend Robot" / "Revoke Robot" buttons (with confirmation dialog).  
- Push notification when own robot receives revocation broadcast.

**Test requirements:**  
- Revoked robot cannot issue commands to peers  
- Peer invalidates cached consent for revoked RRN  
- ESTOP from revoked robot still accepted  
- RESUME from revoked robot rejected  
- Quarantine mode triggered after staleness window

### GAP-05: Consent Wire Protocol

- `rcan/audit.py`: Extend audit writer to record consent events.

**rcan-ts changes:**  
- `src/message.ts`: Add new message types.  
- `src/types.ts`: Add `ConsentRequest`, `ConsentGrant`, `ConsentDeny` interfaces.  
- New `src/consent.ts`: Builder and verification functions.

**OpenCastor changes:**

- `castor/api.py`: Add endpoints `POST /api/consent/request`, `POST /api/consent/respond`.  
- `castor/offline_fallback.py`: Add local SQLite consent store; load on startup if Firestore unreachable.

**RRF changes:**  
- Add `GET /api/v1/consent/{request_id}` endpoint.  
- Add `POST /api/v1/consent` for cross-registry consent requests.  
- Store consent records in Firestore with `consent_type`, `granted_scopes`, `expires_at`.

**opencastor-client changes:**  
- Push notification for incoming consent requests (rich notification with approve/deny actions).  
- Consent management screen: view active consents, revoke access, see expiry.  
- Service consent requests shown with distinct "⚠️ Service" banner.

**Test requirements:**  
- End-to-end consent flow: Robot A requests access to Robot B → owner approves → Robot A commands Robot B  
- Consent grant without valid owner JWT → rejected  
- Expired consent → rejected  
- Offline consent blob: export → import → grant accepted without Firestore

### GAP-06: Offline Operation Mode

- `rcan/safety.py`: Add `offline_mode`, `offline_since_s`, `offline_cross_owner_blocked` to Protocol 66 manifest.  
- New config fields: `offline.grace_period_s` (default 86400), `offline.cross_owner_grace_s` (default 3600), `offline.max_key_ttl_s` (default 86400).

**rcan-ts changes:**  
- `src/registry.ts`: Add `LocalKeyCache` class; fallback to cache in `validateToken()`.  
- `src/safety.ts`: Extend Protocol 66 manifest type.

**OpenCastor changes:**

- `castor/connectivity.py`: Detect registry connectivity; trigger offline mode transitions with audit events.  
- `castor/auth_jwt.py`: Route validation through `LocalKeyCache` when registry unreachable.  
- `castor/conformance.py`: Expose `offline_mode` and `offline_since_s` in `/api/safety/manifest`.

**RRF changes:** Add `GET /api/v1/public-keys` endpoint returning current signing keys for client caching.

**opencastor-client changes:**

### GAP-09: Key Rotation

**Spec section to add:** §8.6 — Key Lifecycle and Rotation  
**Spec text summary:** Robots and operators SHOULD maintain JWKS at `/.well-known/rcan-keys.json` per RFC 7517. Keys MUST include `exp` (max 365-day validity). ### GAP-11: QoS / Delivery Guarantees

**Spec section to add:** §5.3 — Quality of Service  
**Spec text summary:** `qos: 0|1|2` field on all messages. QoS 0 = fire-and-forget. QoS 1 = at-least-once, sender retries ≤ `max_retries` (default 3) with exponential backoff from 100ms. QoS 2 = exactly-once (two-phase: COMMAND_ACK + COMMAND_COMMIT). Safety messages MUST use qos ≥ 1. ESTOP MUST use qos = 2. TELEOP MUST use qos = 0. Failure to ACK safety message after max retries → local safety halt. Receivers MUST send COMMAND_ACK within 500ms for qos ≥ 1.

**rcan-py changes:**  
- `rcan/types.py`: Add `QoSLevel = Literal[0, 1, 2]`; add `qos: QoSLevel` to `RCANMessage`.  
- `rcan/message.py`: Add `COMMAND_ACK=17`, `COMMAND_COMMIT=18` types.  
- New `rcan/qos.py`: `QoSSender` (manages retry logic, exponential backoff, ACK tracking); `QoSReceiver` (sends ACK, manages exactly-once dedup).  
- `rcan/node.py`: Integrate `QoSSender`/`QoSReceiver`; enforce qos minimums by message type; trigger safety halt on ACK timeout for safety messages.  
- `rcan/exceptions.py`: Add `QoSAckTimeoutError`, `SafetyHaltError`.

**rcan-ts changes:**  
- `src/types.ts`: Add `QoSLevel` type.  
- New `src/qos.ts`: `QoSSender`, `QoSReceiver`.

**OpenCastor changes:**  
- `castor/api.py`: Respond with `COMMAND_ACK` for qos ≥ 1 messages within 500ms.  
- `castor/safety/`: Trigger safety halt procedure when ESTOP ACK times out.  
- `castor/channels/`: Ensure all safety message channels enforce qos = 2.

**RRF changes:** None (QoS is transport-layer concern).  
**opencastor-client changes:** Send UI: show delivery confirmation indicator for safety commands. Show "⚠️ ESTOP delivery unconfirmed" alert if ACK not received.

**Test requirements:**  
- QoS 1 message retried on first failure, delivered on second  
- QoS 2 ESTOP: COMMAND_ACK received, then COMMAND_COMMIT  
- ACK timeout for ESTOP triggers local safety halt  
- QoS 0 TELEOP message not retried  
- Safety message with qos=0 → rejected

## Migration Notes

### rcan-py v0.4.x → v0.5.0
- **No breaking changes** in public API. All new parameters have defaults.
- `RCANMessage.from_dict()` now validates `rcan_version` as first step. Messages with MAJOR version mismatch raise `VersionIncompatibleError` (new exception — catch if needed).
- MessageType enum integer values may change for some types (canonicalization). **Any code comparing `msg.msg_type == 5` (by integer) must switch to `msg.msg_type == MessageType.CONFIG`.**
- `SPEC_VERSION` constant moves to `rcan/__version__.py`; import path changes from `rcan.message.SPEC_VERSION` to `rcan.SPEC_VERSION`.

### rcan-ts v0.4.x → v0.5.0
- Same enum integer caveat as rcan-py.
- New required types exported from `src/index.ts`; existing imports unchanged.
- `validateVersionCompat()` is a new required call if you build custom message routing.

### OpenCastor v2026.3.16.x → v2026.3.17.x
- Run Firestore migration script before deploying: adds `revocation_status`, `clock_synchronized`, `consent_records` fields to robot documents with safe defaults.
- `offline_fallback.py` now initializes a local SQLite DB at startup (`~/.castor/consent.db`). First-run creates and populates from Firestore on next sync.
- Replay cache requires ~10MB RAM for 10,000 msg_id entries at default settings. Tune `security.msg_id_cache_size` on low-RAM devices.
- CONFIG safety parameter changes now require `role: creator` JWT. Any automation that was using `role: owner` for safety configs must be updated.

### RRF → v1.5.0
- Firestore security rules updated: new collection paths for consent records, training data consent, robot keys. Deploy `firebase deploy --only firestore:rules` before deploying new Cloud Functions.
- New Cloud Function endpoints: deploy with `firebase deploy --only functions` (no gcloud CLI required).
- No existing data migration required; new fields added with defaults via migration script.

### opencastor-client → v1.5.0
- App update is transparent to end users.
- On first launch after update, users see a "New in v1.5" onboarding sheet covering consent management and privacy dashboard.
- No user data migration required; new data structures populated from API on first use.

## Safety Invariants (Non-Negotiable)

The following constraints from Protocol 66 MUST be preserved across all v1.5 changes:

1. **ESTOP is never rate-limited.** The new replay cache (GAP-03) MUST have a special bypass for `ESTOP` commands where the msg_id check is applied but the rate-limiter is not. Fresh ESTOP (within 10s) always goes through.
2. **ESTOP is never blocked by new security checks.** Key rotation validation, revocation checks, and delegation chain validation MUST NOT block ESTOP receipt. ESTOP is accepted from any identified source; only RESUME has tighter controls.
3. **QoS 2 for ESTOP is receive-side, not send-side gating.** The ESTOP is processed immediately on receipt; COMMAND_ACK is sent back as confirmation, not as a condition for ESTOP execution.
4. **Offline mode does not block owner-level ESTOP.** In offline mode, owner JWT ESTOP from local network is ALWAYS accepted if the cached key matches.
5. **CONFIG safety overrides require creator role.** No delegation chain depth can grant safety parameter override to a non-creator principal. This is a hard check, not scope-based.
6. **`local_supremacy` is inviolable.** mDNS-discovered local peers always discoverable regardless of registry state.
7. **`confidence_gate` values are immutable from outside castor.** No `CONFIG_UPDATE` payload may change confidence gate thresholds — these are compile-time constants in castor.
